# 契約を先に凍結して外部 API を後付けする

契約の中身はこの skill が持たない。単一の真実は `contracts/〈API名〉.md`。毎周これを読み戻す。
コードは読者の環境で生成する成果物。この skill は回し方だけを持つ。
`loop-contract` と併せて使う。周回・予算・人間ゲートはそちらが持ち、この skill は外部 API 連携の上乗せだけを持つ。

## 手順

1. 基準線を取る。後付けを始める前に既存のゲートを走らせ、緑であることを確認する。赤があるなら先に直すか、赤の一覧を記録してから進む。ここを飛ばすと、壊したのか元から壊れていたのか分からなくなる。
2. 境界の契約を草案として書かせる。実装より先。項目は、配布物の `templates/api_contract.md` を `contracts/〈API名〉.md` に写して埋めさせる(この skill は項目を列挙しない)。埋めるときの規律は次の2点。
   - 外部を呼ぶ箇所を一つのアダプタに閉じ、その差し込み口を先に決める。モックも契約ゲートもここに差す。
   - 契約に書かない挙動は「未定義」と明記する。空欄を実装で埋めさせない。
3. 凍結する。人間が草案を確認して承認し、以降の改訂は人間の権限とする。Maker は書き換えない。
4. フォールト注入モックを書かせる。モードは 通常 / 遅延 / 接続失敗 / 4xx(認証・入力不正) / 5xx / レート制限(429 と Retry-After) / 間欠故障。seed を受け取り、失敗が起きる位置と順序を seed だけで決める。時刻はクロックを注入して制御し(実時間の待機に頼らない)、呼び出し回数を記録する(冪等性の検証に要る)。実行順や環境依存の乱数を混ぜない。
5. seed が効いていることを確かめる。ここは実装の前に済ませる。
   - 同じ seed で2回走らせ、失敗系列が一致すること。
   - 違う seed では系列が変わること。これを見ないと、seed を無視して常に同じ系列を返す実装が緑で通る。
6. 契約ゲートを書かせる。契約の各項に一つずつ対応させる。合否は終了コードで返す。失敗系だけでなく成功系も含める。
   - 成功時の出力と、失敗時に返る形が契約どおりであること
   - タイムアウトが契約値で切れること(接続と読み取りを別に)
   - 再試行が契約の上限で止まること
   - 429 の Retry-After を尊重して待つこと
   - 冪等でない操作を再試行しないこと
7. 契約ゲートが落ちることを確かめる。わざと壊した呼び出し側(タイムアウト無し / 上限無しの再試行 / Retry-After 無視 / 冪等でない操作の再試行)を一時的に用意し、対応する項目が赤になることを一つずつ見る。落ちない項目は何も守っていない。確認したら壊した側は捨てる。
8. 本実装をモックに対して緑にする。契約に無い機能を足さない。
9. 手順1と同じコマンドで既存のゲートを再び走らせ、基準線で緑だったものが一つも赤に変わっていないことを確認する。件数の増減ではなく、個々の結果で比べる。
10. 人間ゲート。本番の API キーを使う結合確認は人間が行う。エージェントに本番を叩かせない。

## やってはいけない

- 実装してから契約を書くこと。ゲートを通すために凍結後の契約を書き換えること。
- seed を持たない乱数でモックを作ること(たまに落ちるゲートになり、原因の切り分けができなくなる)。
- 本番 API を叩いて試すこと。本番の鍵をエージェントに渡すこと。
- 冪等でない操作に再試行を付けること。
- 契約ゲートが落ちることを一度も確かめないまま、緑を根拠に完了を宣言すること。

## 報告様式

done / 変更したパス / 基準線と再実行の結果 / seed の一致と差分の確認結果 / 契約ゲートの件数と赤を確認した項目 / 未達。
