# ツール契約を決めてから MCP サーバを回す

サーバの実体はコードだが、エージェントから見えるのは契約だけ。モデルの揺れはコードで抑え込まず、契約で受ける。契約の中身はこの skill が持たない。単一の真実は `contracts/tools.md`。
`loop-contract` と併せて使う。周回・予算・人間ゲートはそちらが持ち、この skill は MCP サーバの上乗せだけを持つ。

## 手順

1. 契約を先に書く。実装より前に、公開するツールごとの契約を書き出して人間に見せる。項目は、配布物の `templates/tool_contract.md` を `contracts/tools.md` に写して埋めさせる(この skill は項目を列挙しない)。埋めるときの規律は次の2点。
   - 失敗時の応答には**次の一手**を入れる。「不正な引数です」で止めない。何をどう直せば通るのかまで書く。引数不正・該当0件・接続不可のそれぞれに書く。
   - 一覧と取得を分ける。〈条件に合う候補を返すツール〉と〈一件の中身を返すツール〉を**別々の2ツール**として立てる。一覧側には返す件数の上限も契約に入れる。
2. 設定の置き場を先に確かめる。**サーバ定義はリポジトリルート直下の `.mcp.json`**。
   - `settings.json` の `mcpServers` に書いても**黙って無視される**。エラーも警告も出ず、ツールが一つも現れないだけ。
   - `.mcp.json` を書き換えたら Claude Code の**再起動**が要る。再起動するまで反映されない。
   - ツールが見えないときは、サーバのコードより先にこの2点を疑う。枠は、配布物の `templates/.mcp.json.example` をリポジトリルートの `.mcp.json` に写して使う。
   - 起動コマンドと引数、接続先や資格情報を渡す環境変数もここに書く。資格情報の値そのものは書かず、コミットもしない。
3. 実装する。**読み取り専用から始める**。書き込み・削除・外部への送信は最初の一周の範囲を越える。契約に無いツールを足さない。
4. ゲートを書く。**サーバを実際に起動して叩く**。起動と停止はゲート自身が持ち、起動済みを前提にしない。読んで確かめる、では判定しない。合否は終了コードで返す。観点は三つ。
   - (a) 構造: ツール一覧に、契約どおりの名前と引数スキーマで出るか。
   - (b) 利用: 正しい引数で呼べて、契約どおりの戻り値の形で返るか。
   - (c) エラー設計: わざと失敗させ、応答に次の一手が含まれるか。
5. ゲートを壊して、本当に落ちることを確かめる。3通り全部やる。
   - スキーマに無い引数を必須にする → (a) が落ちるか。
   - 範囲を1つずらす → (b) が落ちるか。
   - エラー応答から回復手順を抜く → (c) が落ちるか。
   - 落ちない観点があれば、そのゲートは何も見ていない。直してから先へ進む。壊した変更は必ず戻す。
6. 誤用が出たら、契約面から直す。順番は 説明文 → 引数名 → スキーマの制約 → 最後にコード。
   - 呼ぶべきでない場面で呼ばれる: 説明文に「いつ呼ばないか」が書かれていない。
   - 引数を取り違える: 引数名が曖昧。`id` ではなく値の正体が読める名前にする。
   - コードで弾くのは、契約を直しても再現する場合だけ。
7. 人間ゲート。この周が読み取り専用だけなら「該当なし」と記録して先へ進む。書き込み・削除を足す周が来たら、その周とは別に設計する。既定を空回し(dry-run)にし、実行前に人間の承認を挟む。承認を代行しない。

## やってはいけない

- 全文を一度に返す設計。トークンを浪費する。候補一覧と一件取得の二段に割る。
- モデルの誤用をサーバのコードで力ずくで塞ぐこと。まず説明文と引数名を疑う。
- 破壊的ツールを承認なしで公開すること。
- ゲートが落ちることを一度も確かめないまま、緑を根拠にすること。
- `settings.json` にサーバを定義すること。

## 報告様式

done / 変更したパス / ゲートの終了コードと三観点の内訳 / 手順5で落ちなかった観点 / 契約の変更点。
