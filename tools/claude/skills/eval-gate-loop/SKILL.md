# 評価ゲートを立ててから回す

合格基準の数値はこの skill が持たない。単一の真実はプロジェクトルートの `LOOP_CONTRACT.md`。毎周そのファイルを読み直す。
`loop-contract` と併せて使う。周回・予算・人間ゲートはそちらが持ち、この skill は評価固有の上乗せだけを持つ。
評価コードは読者の環境で生成する成果物であり、この skill は手順だけを持つ。契約がまだ無いなら実装を始めず、初版の作成を人間に求める(配布物の `templates/LOOP_CONTRACT.md` を `LOOP_CONTRACT.md` に写して使う)。人間の判断が要る箇所で人間が不在なら、暫定値で埋めて進めず、そこで止まって問う。

## 手順

1. **データを3つに分ける。** 学習(記憶)用・しきい値調整用・評価用(ホールドアウト)に、実装へ触れる前に分割する。どれがどれかを一覧ファイルに書き出し、その所在を `LOOP_CONTRACT.md` の `data:` 欄に書く。良品だけで学習する手法なら、不良は調整用と評価用にのみ置く。分割の比率と偏りの均し方は人間と決めて契約に書き、以後変えない。ホールドアウトは実装が終わるまで**実装する側**の目と文脈に入れない(中身を読まない・並べて眺めない・可視化しない)。ホールドアウトを開くのは評価ゲートのコードだけにする。この順序を崩すと以降の数字は全部無効になり、後から取り返せない。
2. **合格基準を先に数値で決める。** 実装を見る前に、指標・合格ラインの数値・測定条件・運用しきい値の決め方(どのデータで何を規準に選ぶか)を人間と決め、`LOOP_CONTRACT.md` の `done_when` に数値で書く。**どの指標を採るかは題材で変わるので、この skill は決め方だけを持つ**。押さえる観点は三つ、〈全体の分離性能〉〈通すべきものを誤って弾く率の上限〉〈弾くべきものを見逃す率の上限〉。後の二つは運用しきい値を1つに固定した上で測り、**しきい値は調整用データだけで決めてホールドアウトで選び直さない**。しきい値の数値そのものはモデルができてから調整用データで導いてよいが、導き方は後から変えない。当てずっぽうと同じ性能が合格になる基準値を置かない(手順4がすり抜ける)。実装を見てから基準を決めると必ず甘くなる。
3. **評価ゲートを CLI として書かせる。** ゲートを書く役は実装する役と分ける。分けられないなら、ゲートとデータ一覧を実装より先に確定させ、内容のハッシュを契約に控えて以後変更しない。ゲートは評価用のラベルを読むが、ラベルや個々の事例を画面へ出さない(出るのは集計値だけ)。入力は〈評価用データ〉と〈モデルの予測〉、出力は各指標の実測値と基準値の並記。予測の受け渡し形式と、しきい値をゲートへ渡す経路は最初に決めて契約に書き、以後変えない。合否は**終了コード**で返す(全基準を満たせば 0、1つでも下回れば 1)。標準出力の文言で合否を判定しない。件数0や NaN、片方のラベルしか無い入力は、緑ではなく異常終了にする。
4. **ゲート自体を検査する。** 実装を回す前に、わざと壊した予測をゲートへ通し、exit 1 が返ることを確かめる。最低限この3つ: 全件同じスコア、ラベルを反転した予測、乱数のスコア。加えて完全に正しい予測で exit 0 になることも確かめる。ここを飛ばすと「何を入れても 0 を返すゲート」が最後まで緑に見える。
5. **実装を回す。** 実装者はゲートを実行できるが、ゲートも評価用データも書き換えられない(`LOOP_CONTRACT.md` の `guardrails.paths_forbidden` に入れる)。1周ごとに全指標の実測値を1行1周で追記するスコア履歴に残し、その置き場は `LOOP_CONTRACT.md` の `artifacts:` 欄に書く。次周の冒頭で読み直す。前周より悪化したなら、悪化したと書く。
6. **周回上限で止める。** `budget.loops` を使い切ってもゲートが緑にならないなら、基準・手法・データのどれが原因と見ているかを添えて人間に返す。上限を自分で伸ばさない。
7. **人間ゲート。** 全指標が基準を満たした後、誤検知した現物と見逃した現物を人が目で見る。数字が通っても中身が納得できないことはあるので、ここは機械に判定させない。承認を代行しない。

## やってはいけない

- モデルを作った当人に自分の精度を採点させること。ゲートは独立に実行し、終了コードだけを合否とする。兼務するなら毎周ゲートとデータ一覧のハッシュを手順3の控えと突き合わせ、一致しない周の結果は無効にする。
- 評価用データを実装の過程で見せること。しきい値やハイパーパラメータを評価用データへ当て込むのも同じ違反にあたる(見ているのと変わらない)。
- 基準未達を「概ね良好」「実用上は十分」と書き換えて報告すること。落ちたなら、どの指標がいくつ足りないかを数字で書く。
- ゲートの出力文言をパースして合否にすること。文言は人が読むためのもので、判定は終了コードが持つ。
- ゲートが緑にならないので基準を下げること。基準の改訂は人間の権限で、下げた事実と理由を残す。

## 報告様式

done / ゲートの終了コード / 各指標の実測値と基準値 / 手順4の破壊テストの結果 / 残りの周回数 / 未達の項目。
