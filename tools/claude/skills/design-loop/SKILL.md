# 検討ループ

実装ループと違い、合否関数が一つに決まらない。合否は「基準ごとのスコアと重み」で表す。
**基準の定義・重み・採点の尺度は ADR テンプレートにのみ置く。この skill は手順だけを持ち、再定義しない。** 配布物の `templates/ADR.md` を `docs/adr/NNNN-〈要約〉.md` に写して使う。
`loop-contract` と併せて使う。周回・予算・人間ゲートはそちらが持ち、この skill は設計比較の上乗せだけを持つ。

## 手順

1. **契約を凍結する**
   提供層の契約を書き、人間の承認を得る。項目は、配布物の `templates/serving.yaml` を `contracts/serving.yaml` に写して埋める(この skill は項目を列挙しない)。
   以降の全案はこの契約を満たすことが前提。契約が曖昧なまま案を出させない(比較できなくなる)。

2. **評価基準と重みを固定する**
   ADR テンプレートの「評価基準と重み」を先に埋め、人間の承認を得る。
   **案を見てから重みを動かさない**(結論から逆算する後付けになる)。

3. **案を起草する(起草者)**
   同一様式で3案。散文の提案書は並べられないので、必ず次の項目に割って出させる。
   - 各層の方式 / 提供層スキーマの写像 / 前提とリスク / この案が却下されるとしたらどの条件か
   置き場は**案ごとに1ファイル**: `docs/adr/NNNN-〈要約〉/案A.md`(案B, 案C も同様)。ADR 本体へ本文を写さず、比較表と根拠1行だけを載せる。
   起草者には**採点させない**。起草者名を ADR の `起草者` 欄に書く。

4. **サンプルでゲートに掛ける**
   隔離したサンドボックスで、本番へ接続せず実データも移さずに判定する。合否は終了コードで返す。
   - 契約適合: 旧契約と案のスキーマを比べ、列の削除や型の変更があれば
     非ゼロで落ちるゲートを作って回す(無ければこの周で作らせる)
   - 鮮度試算(想定ロード頻度でSLAを満たすか) / コスト概算(保管量と走査量から月次)
   ここで落ちた案は比較表に載せない。

5. **採点する(採点者)**
   ゲートを通過した案だけを、起草者とは別指示のエージェントが基準ごとに採点する。**採点の尺度は ADR テンプレートに従う**(この skill は尺度を持たない)。採点者名を ADR の `採点者` 欄に書く。
   根拠のないスコアは差し戻す。加重合計は比較の補助であって、決定そのものではない。

6. **人間が決める**
   比較表を人間が読み、重み配分の妥当性を承認して1案を選ぶ。**唯一の人間ゲートで、機械が代替しない。**

7. **ADR に外部化する**
   ADR テンプレートを写して埋める。**却下案と却下理由と復活条件を必ず書く**。
   `起草者` / `採点者` / `周回` も埋める(後から停止条件を検証できるように)。**起草者と採点者が同一名の ADR は差し戻す**。
   検討ループの価値は、次の周回で同じ案を再検討しないことにある。却下欄が空の ADR は差し戻す。

## 停止条件

- 周回上限は `LOOP_CONTRACT.md` の `budget.loops` に従う(`周回` 欄で数える)。使い切ったら停止し、人間に重み配分の再検討を求める(案を増やし続けない)。
- 全案がゲートで落ちたら、案ではなく**契約か制約が矛盾している**。契約凍結(手順1)へ戻る。
- 加重合計が僅差なら、機械は決めない(何点差を僅差とみなすかは ADR の評価基準に書く)。何を重く見るかの判断を人間へ渡す。

## やってはいけない

- 周回の途中で基準や重みを変える(出したい結論に合わせた逆算になる)。
- 起草者に自己採点させる(自案びいきが入る)。
- 実データをサンドボックスへ持ち込む(PII 列は契約の `governance.pii_columns` を見る)。
- 契約の非互換変更を黙って通す(契約適合のゲートが落ちたら、再凍結の承認へ回す)。
- 制約違反の案を減点で通す。**制約は足切り**で、触れた案は採点せず ADR の「採点前に除外した案」へ落とす。

## 報告様式

done / ADR のパス / ゲートの終了コードと落ちた案 / 起草者と採点者の名前 / 使った周回数 / 人間の決定待ちか否か。
