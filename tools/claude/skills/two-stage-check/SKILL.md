# 二段構えで受け入れる

判定を機械と人間に切り分けるのがこの手順の全て。
切り分けが曖昧だと、機械に「見やすさ」を判定させるか、人間に数値の検算をさせるかのどちらかが起きる。
人間ゲートの項目の中身はこの skill が持たない。単一の真実は `docs/ui_human_gate.md`。
`loop-contract` と併せて使う。周回・予算・人間ゲートで止まる点はそちらが持ち、この skill は画面受入の上乗せだけを持つ。

## 手順

1. **指標を先に固める**。何を、どの粒度で、どの単位で、いつ時点の値として出すか。併せて、突合先となる元データの所在を確定させる(ここが未定なら機械ゲートは正しさを判定できない)。一覧にして、その画面を毎日使う現場の代表に確認を取る。空欄のまま画面を作り始めない。
2. **判定項目を仕分ける**。基準はひとつ、〈同じ入力を与えたら毎回同じ答えが出るか〉。出るなら機械(数値の正しさ、合計と内訳の整合、閾値どおりの色分け、コントラスト比、要素のはみ出しと重なり、単位と更新時刻の表示有無)。出ないなら人間(見やすさ、視線の流れ、現場で通じるか)。迷う項目は分割する。「比が基準値以上か」は機械、「その照明と距離で読めたか」は人間。同じ対象でも、測れる量と確かめる体験は別項目である。仕分け表は1ファイルに書き出し、置き場を `LOOP_CONTRACT.md` の `artifacts:` 欄に書く。人間に見せてから先へ進む。
3. **機械ゲートを書く**。合否は終了コードで返す(合格 0 / 不合格 1)。判定対象は元データと、描画後の DOM および算出スタイル(座標、寸法、色)。画像を見て判断する経路は作らない。
4. **機械ゲートに、自分が判定していない項目を出力させる**。合格時にも必ず出す。「このゲートはコントラスト比と数値整合だけを見た。見やすさと視線の流れは判定していない」と明示する。これを省くと、緑を見た人間ほど「全部見てもらえた」と受け取る。
5. **人間ゲートを作る**。配布物の `templates/ui_human_gate.md` を `docs/ui_human_gate.md` に写して使う。枠がまだ無ければ置いてもらうよう人間に求め、項目を自前で発明しない。機械が同じ問いを済ませた項目は載せない(重複は人の時間を奪う)。各項目は「はい / いいえ」で答えられる問いにする。「見やすいか」ではなく「2m 離れて主要な数値を読めたか」のように、その場でやってみれば答えが出る形にする。感想を書く欄は作らない。
6. **周回する**。機械ゲートが赤のうちは Maker の中で直す。緑になったものだけを人間ゲートへ出す。人間ゲートは最後の一回。いいえが 1 つでもあれば、その項目だけを差し戻して手順3へ戻る。ただし原因が指標の選び方や粒度そのものにあるなら、画面を直さず手順1へ戻る。

## やってはいけない

- 機械に「見やすい」「わかりやすい」「バランスが良い」を判定させること。
- スクリーンショットを見て良し悪しを述べ、それを機械ゲートと呼ぶこと。目視は人間ゲートの仕事。
- 機械ゲートの守備範囲を黙ったまま緑だけを報告すること。
- 人間ゲートに数値の検算をさせること。人の時間を機械の仕事に使わせない。
- 機械ゲートが赤のまま人間ゲートを開くこと。

## 報告様式

done / 仕分け表のパス / 機械ゲートの終了コードと、通した検査の件数(指標数ではなく検査数) / 機械が判定していない項目 / 人間ゲート待ちか否か。
