# commit-message

ステージ済みの変更(`git diff --cached`)を読み、Conventional Commits 規約に沿った
コミットメッセージを1件生成する。生成のみを行い、コミットの実行やプッシュは行わない。

## 形式

    <型>[任意の scope]: <要約行>

    <本文(理由と背景)>

    <フッタ(BREAKING CHANGE / 関連 Issue)>

- 型・scope・要約行は1行目にまとめる。scope は変更範囲を表す任意の名詞(例: `api`, `ui`)。
- 型と本文・本文とフッタの間はそれぞれ空行で区切る。

## 型の選択

| 型 | 用途 |
| --- | --- |
| feat | ユーザに見える機能の追加 |
| fix | バグ修正 |
| docs | ドキュメントのみの変更 |
| style | 挙動を変えない整形(空白・整列など) |
| refactor | 挙動を変えない内部改善 |
| perf | 性能改善 |
| test | テストの追加・修正 |
| build | ビルド・依存関係の変更 |
| ci | CI 設定・スクリプトの変更 |
| chore | 上記以外の雑務 |

変更が複数の型にまたがる場合は、最も影響の大きい1つを主にし、必要ならコミットを分割する。

## 要約行(50字以内)

- 命令形・現在形で書く(例:「追加する」ではなく「追加」)。文末に句点を付けない。
- 「何を」変えたかを簡潔に。「なぜ」は本文へ回す。
- 50字を超える場合は要約が大きすぎる合図。コミットの分割を検討する。

## 本文(理由と背景)

- 変更の動機(なぜ必要か)と、以前の挙動との差分を書く。実装の逐語説明は避ける。
- 1行はおおむね72字で折り返す。箇条書き可。

## 破壊的変更(BREAKING CHANGE)

- 後方互換を壊す変更は、フッタに `BREAKING CHANGE: <内容と移行方法>` を明記する。
- 併せて型のあとに `!` を付けてよい(例: `feat!: ...`)。silent な破壊を残さない。

## 関連 Issue の参照

- フッタに `Refs: #123` / `Closes: #123` の形で関連 Issue を参照する。
- 複数ある場合は行を分けて列挙する。

## 保存と呼出

- 保存パス: `.claude/skills/commit-message/SKILL.md`
- 呼出(自動): セッションで description がコミット作成タスクに自動マッチ
- 呼出(明示): `/commit-message`
