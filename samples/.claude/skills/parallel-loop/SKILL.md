# parallel-loop

作る者(Implementer)と検める者(Verifier)を分け、検証律速を制御しつつ品質を担保する
並列オーケストレーション。Orchestrator(親)は読まない・書かない。結論とパスだけ受ける。

## 手順
1. **WBS 分解(Orchestrator)**: 案件を独立テスト可能な単位へ分解し、各単位に「ファイル所有権」を割当てる。共有 hotspot は単一所有者にするか、interface を先に凍結する。
2. **claim / 契約**: 各エージェントは開始時に担当ファイル範囲を WBS 上で claim し、schema/interface を先に固定する。部分文脈による重複・矛盾を未然に防ぐ。
3. **worktree 並列起動**: コード書換は `isolation: worktree` で N 本同時起動(実用 4〜8)。各 worktree にタスク特化 CLAUDE.md でスコープを限定。DB/UI 依存は直列レーンへ回す。
4. **各自で3点ゲート**: マージ前に各 worktree で lint → 型チェック → 関連テストを通す(`/verify-gate`)。lint/型は並列安全、テスト全体は直列レーン。
5. **順次マージ**: ゲートを通った worktree から一本ずつマージ。並列のままぶつけず、順序を確定させて統合する。
6. **統合検証(Verifier)**: 統合後に意味整合を確認(自己採点禁止の Maker-Checker)。DB/UI を使う検証はここで直列に実施。
7. **人間承認**: 最終承認ゲートで人間が受理。進捗は WBS へ追記され、Orchestrator が次周回の俯瞰に使う。

## 規律
- 独立単位が3未満なら並列の固定費が割高。直列へフォールバックする。
- 同一ファイルを2つの並列単位が同時編集しない(file ownership)。
- 子の戻りは数行(done / 変更パス / ゲート結果 / 残課題)。差分・ログ・本文は返させない。
